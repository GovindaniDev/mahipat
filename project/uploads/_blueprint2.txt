[Heading1] 1. Product Intent and Strict Build Philosophy
The system must be built as a workflow-driven NGO Operating System, not as a basic donation website. The purpose is to manage donors, donation records, tax-compliance workflows, certificate delivery, fundraising attribution, CA-ready reports, communication history, and actionable donor intelligence from a single controlled platform.
Principle
Developer meaning
Non-negotiable output
Donor-first
Donor is the primary entity. Donations, receipts, 10BE, communications and fundraising attribution attach to donor journey.
Every donor profile must show complete 360-degree history.
Compliance-safe
10BD/10BE cannot work like simple file upload-send. It needs validation, matching, approval and audit.
No certificate can be dispatched without matching and dispatch log.
Attribution-ready
Every donation should know the source: direct, campaign, fundraiser, QR, agency, telecaller, manual entry.
Manual donation entry must include source and optional fundraiser selection.
Audit-ready
Exports, edits, imports, certificate sends, manual matches and financial actions must be logged.
No silent edits for compliance/financial data.
Dynamic and simple
Masters, menus, templates, bank accounts, donor types and alerts should be configurable.
Avoid hard-coded names and one-off screens.
Developer independence
Developers must not need founder explanations for fields and workflows.
Every module must contain fields, validations, buttons, statuses and edge cases.

[Heading2] 24.1 Security Rules
[ListBullet] PAN must be uppercase and masked where full value is not required.
[ListBullet] Only authorized admin can export PAN/compliance data.
[ListBullet] Passwordless update links must be tokenized, time-limited, single purpose and logged.
[ListBullet] Generated PDF links must be signed/non-guessable.
[ListBullet] Every API must enforce tenant isolation and permission checks.
[ListBullet] File uploads must validate MIME/type/size and reject unsafe ZIP contents.
[ListBullet] Manual match requires reason and audit log.
[ListBullet] Delete should be soft delete for financial/compliance records.
