Govindani NGO OS - All User Type Options Blueprint
Role-wise dashboards, menus, buttons, permissions, restrictions, audit rules and SaaS plan visibility
Document Purpose: This file defines exactly what each user type can see and do inside Govindani NGO OS. Developers must implement role-based routing, tenant isolation, plan locks, audit logs, and visible upgrade prompts exactly as described. This document extends the Super Admin options blueprint to every user type in the system.
1. Non-Negotiable Role Architecture
Every login must resolve role, tenant, plan, permissions and add-on status before dashboard render.
Every user type gets a separate dashboard experience. Do not dump all users into one admin panel.
All modules remain visible where commercially useful, but locked modules must show upgrade motivation, not dead errors.
Tenant users cannot access another NGO tenant. Internal Govindani users require explicit support/admin permissions.
Sensitive exports like PAN, 10BD, 10BE, bank reconciliation and compliance files require separate export permission.
Workflow
Required System Behaviour
Login Routing
After login, identify tenant_id, role_id, plan_id, permission set, add-on status, and route to role-specific dashboard.
Locked Feature Display
Show locked modules with blur/lock card, plan required, benefit statement, Upgrade CTA. Do not hide feature cards.
Audit Logging
Every create/edit/delete/export/send/approve/impersonate/payment action creates audit log with user, role, tenant, timestamp, IP/device, old/new data where applicable.
Data Isolation
Every query must be tenant-scoped unless Govindani Super Admin internal view is used.
Soft Delete
Financial, donation, receipt, compliance, membership payment, payout, and audit records are never hard deleted by tenant users.
Notification
Role-specific notifications are sent by in-app alert, email, WhatsApp/SMS where enabled and paid.

4. Global UI Rules for All User Types
Sidebar must show only relevant direct menus plus locked upgrade cards where plan upsell is intended.
Every dashboard top bar must show NGO name, current role, current plan, notification bell, help/ticket button, and profile menu.
Every action button must either perform action, open modal/drawer, redirect to exact page, or display plan lock modal.
Every export button must check export permission and log filters, row count, file type, timestamp and user.
Every delete action must be labelled Archive/Deactivate/Soft Delete unless it is non-statutory content like draft blog or unused template.

8. API Guard Rules
All APIs must check tenant_id, role permission and plan/add-on access.
Write APIs must create audit logs for financial, compliance, membership, user, website, campaign and payout actions.
Export APIs must validate export permission separately from view permission.
Impersonation APIs must never bypass audit logging. UI must show impersonation banner.
Locked-plan APIs return 402/plan_required style response with required_plan and upgrade_url, not generic 403.

10. Final Developer Instruction
Do not build one universal admin dashboard. Build role-specific dashboards, permission-scoped menus, tenant-scoped data, plan-lock states, and audit-logged actions. If a developer cannot explain what a user type can see, click, export, approve, delete, or request, the implementation is incomplete.
