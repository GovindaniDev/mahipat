Primary build rule
Developers must build from this document without founder dependency. Any ambiguity must be logged before coding.
This document is intentionally strict. The goal is not to create a beautiful demo only. The goal is to create a real SaaS operating system for Indian NGOs where donations, compliance, donor data, roles, billing, website, fundraisers, agencies, volunteers and CA reporting all connect through one controlled data model.
CA and compliance disclaimer: Government forms, tax rates, statutory due dates and portal formats can change. The product must store template mappings and configurable compliance rules. No in-app text should claim legal finality unless verified by a practising CA and the current government utility.

[Heading1] 1. Product Definition and Non-Negotiable Build Rules
Govindani NGO OS is India-first and must support NGOs, Trusts, Temples, Gaushalas, Schools, Foundations, religious organizations and donation-driven bodies. Later internationalization must be architecturally possible, but v1 must deeply support Indian workflows: PAN, 80G, 12A/12AB, 10BD, 10BE, FCRA toggle, CSR, donor receipts, CA exports, GST invoices for SaaS billing, Razorpay/Cashfree subscription billing, custom domains and role-based access.
Rule
Developer meaning
Failure if ignored
Multi-tenant first
Every table carrying NGO business data must include tenant_id.
One NGO can see another NGO data; fatal breach.
Role + permission driven
Role name is not enough. Every action must check permission key and tenant scope.
CA/telecaller/fundraiser gets wrong access.
Donor is the core entity
Donation, receipt, 10BD, 10BE, WhatsApp, email, fundraising attribution attach to donor.
Reports break and compliance becomes manual Excel.
Compliance snapshots
Receipt/10BD/10BE exports freeze data at time of generation. Later profile edits do not silently rewrite past PDFs.
Audit failure and CA cannot trust records.
Plan gating is visual + backend
Locked modules can be visible as blurred cards, but API must still deny access.
A user can bypass lock from browser.
No hard delete for financial/compliance records
Use soft delete, void, reverse, restore and audit. Permanent delete allowed only for failed onboarding/test tenants by Super Admin.
Financial history disappears.
Configurable laws/templates
10BD/10BE/GST/TDS templates are versioned settings.
Government utility changes will break product.
Prototype must simulate real product
Demo screens must have real roles, realistic data and state transitions.
Developers build a pretty but useless mockup.
[Heading1] 2. Current Prototype Gaps and Corrected Direction
[ListBullet] Current HTML is visually usable but still behaves like a single-user static admin panel.
[ListBullet] Existing fake login must be replaced by role-based demo login accounts and route-specific dashboards.
[ListBullet] Plan selection, locked modules, subscription state, add-ons, and usage limits must be visible in the prototype.
[ListBullet] The fundraiser/agency/telecaller hierarchy is not optional; it is a core differentiator and must be treated as ERP.
[ListBullet] The CA department must not be a simple reports page. It must be a statutory command center with validation queues, filing packs and downloadable registers.
[ListBullet] Every button in compliance, billing, donation and payout modules must create an audit log event.

[Heading1] 7. Billing, Invoice, Add-On and Overage Rules

Rule

Razorpay and Cashfree supported. Provider config is Govindani-side for SaaS billing; NGO connects own Razorpay for donations.

Excess receipts enter HELD state. One overage payment releases generation/sending/download for those receipts; no double charge.

[Heading1] 14.1 Fundraiser Campaign Assignment Flow
[ListNumber] NGO Admin creates campaign/cause/case.
[ListNumber] Admin toggles Enable Fundraising = Yes.
[ListNumber] Admin selects assignment targets: individual fundraisers, agency owners, telecaller groups, fundraiser groups, or all active fundraisers.
[ListNumber] Admin selects commission rule: per-donation, lifetime donor, first donation only, recurring included yes/no, attribution validity days, exclude refunded donations.
[ListNumber] System generates tracking links and QR codes per target. No duplicate campaign is created.
[ListNumber] Target users receive in-app notification, email and WhatsApp if enabled.
[ListNumber] Agency Owner sees new campaign and distributes to telecallers manually or auto-distribute.
[ListNumber] Telecaller/fundraiser uses link/QR. Donation stores tracking_id, fundraiser_id, agency_id, telecaller_id, campaign_id and attribution snapshot.
[ListNumber] Dashboard updates raised amount, eligible commission, pending payout and validation issues.
[ListNumber] Fundraiser/agency can raise withdrawal request. Admin reviews, marks paid/not paid/remind later. No automatic money transfer in v1.

NGO creates campaign -> Enables fundraising -> Assigns fundraiser/agency/telecaller group -> Tracking links/QR generated -> Agency distributes to telecallers -> Donor donation captured with attribution -> Commission ledger calculated -> Withdrawal request -> Admin approval -> Mark paid/not payable/remind -> Email/payout audit -> CA payout/TDS report
Diagram 4: Volunteer/event lifecycle
NGO creates event -> Volunteers see event -> Register/waitlist -> Check-in attendance -> Hours calculated -> Task completion -> Certificate eligibility -> Certificate generated -> Volunteer report

[ListBullet] PAN masked except permitted internal handling.
[ListBullet] Backdated donation allowed only to permitted roles.
[ListBullet] NGO owner can edit generated receipt-linked donation only with audit.
[ListBullet] CA dashboard export permissions configurable.
[ListBullet] Super Admin impersonation logged.

[Heading1] 31. SaaS Scale Architecture
Target design should support thousands of NGOs without rewriting core modules. Start as modular monolith, but isolate services by package and database boundaries so modules can later split.
Scale concern
Decision
Tenant isolation
Every business table has tenant_id; query filters mandatory; add database constraints where possible.
Object storage
Receipts, certificates, uploads, exports stored privately with tenant path and signed URLs.
Queue workers
Email, WhatsApp, SMS, PDF generation, exports, imports and ZIP processing run async.
Rate limits
API keys, donor update tokens, OTP, WordPress plugin endpoints and public donation forms must have rate limits.
Large imports
Above 25,000 rows route to admin/developer queue; process through background job with validation report.
Exports
Export job table, status, checksum, retention, permission snapshot.
Audit
Append-only audit logs; do not allow user edits.
Configurable compliance
Government format mappings stored in versioned templates, not hard-coded.
Feature flags
Plan entitlements and tenant-specific toggles drive UI and API access.
Impersonation
Super Admin impersonation must be explicit, logged and visually bannered.

[Heading1] 34. Final Instruction to Developers
Do not build scattered pages. Build a controlled multi-tenant NGO SaaS OS. The system must have tenant isolation, role routing, plan entitlement, audit logs, donor identity, donation ledger, compliance validation, communication logs and billing state from the first prototype. If a developer builds a screen without knowing which role can see it, which plan unlocks it, which table stores it, which audit event logs it and which report consumes it, the work is incomplete.
